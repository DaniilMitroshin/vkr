# Архитектура проекта

## Общий подход

Проект реализован как монолитное клиент-серверное приложение на Go. Внутри одного исполняемого файла объединены REST API, Telegram-бот, бизнес-логика и слой доступа к данным. Система использует PostgreSQL как основное хранилище.

## Уровень представления

1. Telegram-бот — основной интерфейс студента и администратора: `internal/bot/bot.go`.
2. REST API — дополнительный интерфейс для проверки, импорта, экспорта и интеграций: `internal/httpapi/server.go`.

## Уровень прикладной логики

Пакет `internal/service` реализует прикладные операции: импорт файлов, регистрацию, получение выборов, отправку выбора, экспорт CSV/JSON и генерацию DOCX. Сервис работает через интерфейс `Store`, что отделяет бизнес-операции от конкретной реализации PostgreSQL.

## Уровень доступа к данным

Пакет `internal/repository` содержит реализацию работы с PostgreSQL на базе `pgxpool`: миграции, импорт, upsert-операции, выборки, сохранение записей, проверку правил выбора и автоматическое распределение.

## Файловый слой

Файловый слой используется для:

- чтения входных CSV/JSON-файлов при импорте;
- хранения SQL-миграций в `migrations/`;
- копирования стартовых файлов в Docker-образ;
- формирования DOCX-файла в памяти.

## База данных

PostgreSQL хранит студентов, выборы, группы, варианты дисциплин, записи студентов, администраторов и список примененных миграций.

## Контейнеризация

`docker-compose.yml` описывает два сервиса:

- `postgres` — PostgreSQL 16 с volume `postgres_data`;
- `app` — Go-приложение, собираемое из `Dockerfile`.

`Dockerfile` использует multi-stage build: сначала собирается Go-бинарник, затем он копируется в минимальный Alpine-образ вместе с `migrations/` и `input/`.
