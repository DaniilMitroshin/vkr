РЕФЕРАТ

Выпускная квалификационная работа посвящена разработке клиент-серверного приложения для автоматизации процесса выбора и распределения студентов по дисциплинам по выбору. Актуальность работы обусловлена тем, что в образовательной организации процесс выбора дисциплин часто связан с ручным сбором заявок, последующей обработкой таблиц, контролем лимитов мест и подготовкой отчетных документов. При увеличении количества студентов, групп и дисциплин такой процесс становится трудоемким и подверженным ошибкам.

В рамках работы разработан программный прототип на языке Go. В качестве основного пользовательского интерфейса используется Telegram-бот, а для проверки и интеграции предусмотрен REST API. Данные хранятся в PostgreSQL, запуск приложения и базы данных выполняется с использованием Docker Compose. Система поддерживает импорт студентов и дисциплин из CSV/JSON, выбор дисциплин студентом, административные команды, экспорт результатов в CSV/JSON и генерацию заявления студента в формате DOCX.

В работе рассмотрены предметная область, требования к системе, архитектура приложения, структура базы данных, алгоритм автоматического распределения, реализация основных компонентов и результаты тестирования. Разработанное приложение может использоваться как основа для автоматизации учебного процесса, связанного с выбором дисциплин.

Ключевые слова: клиент-серверное приложение, Go, PostgreSQL, Telegram Bot API, Docker, Docker Compose, CSV, JSON, DOCX, дисциплины по выбору, распределение студентов.

ABSTRACT

The graduation qualification work is devoted to the development of a client-server application for automating the selection and distribution of students among elective courses. The relevance of the work is determined by the need to reduce manual processing of applications, control course capacity limits and prepare resulting documents.

The implemented prototype is written in Go. The main user interface is a Telegram bot, while a REST API is provided for testing and integration purposes. PostgreSQL is used for persistent storage, and Docker Compose is used to run the application and database. The system supports CSV/JSON import, student course selection, administrator commands, CSV/JSON export and DOCX statement generation.

The work describes the domain area, system requirements, application architecture, database schema, automatic assignment algorithm, implementation details and testing results.

ВВЕДЕНИЕ

В современных образовательных программах дисциплины по выбору позволяют студентам формировать индивидуальную траекторию обучения. Такие дисциплины дают возможность учитывать интересы обучающихся, особенности образовательной программы и потребности рынка труда. Однако организация выбора дисциплин требует учета большого числа факторов: состава учебных групп, доступности дисциплин для конкретных направлений, сроков подачи заявок, лимитов мест, особенностей обязательных дисциплин по выбору, факультативов и модулей мобильности.

На практике сбор заявок часто выполняется с использованием таблиц, форм и ручной обработки данных. Такой подход подходит для небольшого количества студентов, но плохо масштабируется при росте числа групп и вариантов выбора. Администратору необходимо контролировать корректность заполнения данных, отслеживать дедлайны, проверять лимиты мест, формировать итоговые списки и готовить документы. Ошибки на любом из этих этапов могут привести к некорректному распределению студентов или необходимости повторной обработки заявок.

Объектом исследования является процесс выбора и распределения студентов по дисциплинам по выбору в образовательной организации. Предметом исследования являются программные средства автоматизации сбора заявок, проверки ограничений и формирования результатов распределения.

Целью выпускной квалификационной работы является разработка клиент-серверного приложения для автоматизации распределения студентов по дисциплинам по выбору с использованием языка Go.

Для достижения поставленной цели были определены следующие задачи: проанализировать предметную область и существующие подходы к организации выбора дисциплин; определить функциональные и нефункциональные требования к системе; спроектировать архитектуру приложения и структуру базы данных; реализовать серверную часть, Telegram-бота, импорт и экспорт данных; реализовать механизм автоматического распределения студентов по обязательным дисциплинам по выбору; подготовить контейнеризацию приложения и провести тестирование основных сценариев.

Практическая значимость работы заключается в том, что разработанное приложение позволяет сократить объем ручной работы администратора, централизовать хранение заявок, обеспечить проверку ограничений и формировать итоговые выгрузки. Система может быть использована как прототип для дальнейшего внедрения или развития в рамках образовательной организации.

Выпускная квалификационная работа состоит из введения, семи глав, заключения, списка использованных источников и приложений. В первой главе рассматривается предметная область. Во второй главе формулируются требования. В третьей главе описывается проектирование системы. В четвертой главе обосновывается выбор технологий. В пятой главе приводятся сведения о реализации. В шестой главе описывается тестирование. В седьмой главе рассматриваются результаты и перспективы развития.

ГЛАВА 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

Процесс выбора дисциплин является частью организации учебного процесса. Он связывает образовательную программу, учебные планы, группы студентов и доступные дисциплины. Для корректной организации выбора необходимо предоставить студентам информацию о доступных вариантах, принять заявки, проверить ограничения и сформировать итоговые списки. В отличие от произвольного анкетирования, выбор дисциплин имеет последствия для учебного плана студента, поэтому требует точности и воспроизводимости результата.

Ручная обработка заявок обычно строится вокруг электронных таблиц. Администратор готовит список студентов, список дисциплин, рассылает форму выбора, затем переносит данные в таблицы и проверяет ограничения. Такой способ прост и не требует разработки специализированной системы, но имеет существенные ограничения. Данные оказываются распределены между разными файлами, формами и сообщениями, а контроль корректности часто выполняется вручную.

При большом количестве студентов возрастает риск ошибок: студент может выбрать недоступную дисциплину, подать заявку после дедлайна, выбрать больше дисциплин, чем требуется, или попасть в вариант, где уже нет мест. Кроме того, при ручной обработке сложно быстро получить актуальное состояние заполненности дисциплин и сформировать итоговые документы для всех участников процесса.

В качестве распространенных инструментов можно рассмотреть Google Forms, Excel или Google Sheets, LMS-системы и ручную обработку заявлений. Google Forms удобен для сбора ответов, но не является системой распределения и не содержит встроенной предметной модели дисциплин, групп, мест и типов выбора. Excel и Google Sheets позволяют гибко работать с таблицами, однако требуют ручной настройки формул и контроля целостности данных. LMS-системы могут использоваться в учебном процессе, но часто не покрывают специфический сценарий централизованного распределения по дисциплинам с учетом локальных правил образовательной программы.

Для рассматриваемой задачи требуется не просто собрать ответы, а обеспечить последовательный процесс: импорт исходных данных, регистрацию студента, отображение только доступных выборов, сохранение заявки, автоматическое дораспределение студентов без выбора и экспорт результата. Поэтому целесообразно разработать специализированное клиент-серверное приложение, в котором предметные правила будут реализованы на уровне бизнес-логики и базы данных.

Вывод по главе: анализ предметной области показывает, что задача выбора дисциплин требует специализированной автоматизации. Универсальные инструменты сбора форм и таблицы удобны как вспомогательные средства, но не обеспечивают полного контроля ограничений, централизованного хранения данных и автоматического формирования итоговых результатов.

Особенность рассматриваемого процесса состоит в том, что выбор дисциплины не является изолированным действием одного студента. Каждый выбор влияет на заполненность конкретного варианта дисциплины и, следовательно, на возможность выбора этой дисциплины другими студентами. Поэтому система должна рассматривать заявки как общую совокупность данных, а не как независимые ответы анкеты.

При ручной организации процесса администратор обычно вынужден объединять данные из нескольких источников. Один файл может содержать контингент студентов, другой — перечень дисциплин, третий — ответы формы. Далее требуется сопоставить ФИО, группы и выбранные дисциплины. Даже небольшие расхождения в написании ФИО или кода группы могут привести к ошибкам сопоставления. По этой причине в разработанной системе предусмотрена нормализация ФИО и группы при импорте.

Для образовательной организации важна воспроизводимость результата. Если результат распределения был получен вручную, сложно доказать, какие правила применялись в конкретный момент времени. В программной системе правила закрепляются в коде и в ограничениях базы данных. Это делает процесс более прозрачным: можно проверить, какие данные были загружены, какие записи созданы и каким источником они были сформированы.

Отдельной проблемой является подготовка заявлений. Если студент выбирает несколько дисциплин, администратору требуется сформировать документ, отражающий этот выбор. В ручном режиме такие документы часто готовятся по шаблонам и требуют дополнительной проверки. В разработанном проекте предусмотрено формирование DOCX-заявления на основе записей студента, что уменьшает объем ручной работы.

Использование Telegram-бота как клиентского интерфейса также связано с особенностями предметной области. Студенту не требуется устанавливать отдельное приложение или изучать новый веб-интерфейс. Взаимодействие строится через команды и кнопки, а серверная часть при этом сохраняет все данные централизованно. Такой подход особенно удобен для прототипа, где важно быстро проверить основной сценарий.

ГЛАВА 2. ТРЕБОВАНИЯ К РАЗРАБАТЫВАЕМОЙ СИСТЕМЕ

Требования к системе сформированы на основании фактически реализованного программного проекта. Основными ролями являются студент и администратор. Студент взаимодействует с системой через Telegram-бота. Администратор использует Telegram-команды и REST API для загрузки данных, управления выбором и выгрузки результатов.

Для студента реализованы следующие сценарии: регистрация по ФИО и коду группы, просмотр доступных выборов, выбор дисциплин, просмотр текущих записей и получение заявления в формате DOCX. Регистрация не является полноценной авторизацией по логину и паролю. В системе используется идентификация через Telegram ID, который связывается с записью студента после проверки ФИО и группы.

Для администратора реализованы сценарии загрузки студентов и дисциплин, просмотра первых записей студентов, запуска автоматического распределения для обязательного выбора, ручного изменения выбора студента, выгрузки результатов и управления списком администраторов. Права администратора определяются по Telegram ID: начальный список задается переменной окружения, а затем может храниться в базе данных.

К функциональным требованиям относятся: импорт студентов из CSV/JSON; импорт выборов и вариантов дисциплин из CSV/JSON; хранение данных в PostgreSQL; отображение студенту только тех выборов, которые доступны его группе; учет дедлайна при студенческой подаче выбора; контроль лимитов мест; поддержка типов выбора required_choice, elective и mobility; экспорт данных в CSV/JSON; генерация заявления DOCX; наличие REST API и Telegram-интерфейса.

Отдельные правила зависят от типа выбора. Для required_choice студент может выбрать обязательную дисциплину один раз, после чего повторное изменение студентом блокируется. Администратор сохраняет возможность ручной корректировки. Для elective реализован множественный выбор без ограничения по значениям min_selected и max_selected. Для mobility проверяется сумма зачетных единиц в диапазоне, заданном параметрами выбора.

Нефункциональные требования отражают способ эксплуатации системы. Приложение должно запускаться в контейнере Docker, база данных должна работать в отдельном контейнере PostgreSQL, конфигурация должна задаваться через переменные окружения, данные должны сохраняться в Docker volume, а работоспособность HTTP-сервера должна проверяться через endpoint /health.

Входными данными являются CSV/JSON-файлы со студентами и дисциплинами, команды Telegram-бота и JSON-запросы REST API. В файлах студентов используются ФИО, направление и группа. В файлах дисциплин используются код выбора, название выбора, тип, образовательная программа, руководитель, группы, дедлайн, параметры выбора, название дисциплины, количество мест, кредиты, семестр, преподаватель и ссылка.

Выходными данными являются CSV/JSON-выгрузки результатов распределения, списков студентов и зарегистрированных студентов, сообщения Telegram-бота и заявление студента в формате DOCX. Экспорт Excel и генерация PDF в текущей версии кода не реализованы и могут быть рассмотрены как направления дальнейшего развития.

Вывод по главе: требования фиксируют минимально необходимый набор функций для автоматизации процесса выбора дисциплин и соответствуют фактически реализованной системе. Ограничения текущей версии явно выделены, чтобы не приписывать проекту несуществующие возможности.

Требование к регистрации студента связано с необходимостью сопоставить пользователя Telegram с записью в загруженном контингенте. Система не создает произвольного студента при регистрации через бот. Вместо этого она ищет уже импортированную запись по нормализованным ФИО и группе. Такой подход снижает риск появления лишних пользователей и соответствует сценарию, в котором список студентов заранее известен.

Требование к фильтрации выборов по группе необходимо для того, чтобы студент видел только те дисциплины, которые относятся к его образовательной программе или учебной группе. В базе данных это реализовано через отдельную таблицу связи выбора и группы. Такой вариант проектирования позволяет одному выбору быть доступным сразу нескольким группам без дублирования самой сущности выбора.

Требование к контролю дедлайна применяется к студенческому выбору. Если студент пытается отправить заявку после установленного срока, система должна отклонить операцию. При этом административное изменение выбора выполняется отдельным методом и не использует тот же запрет по дедлайну, поскольку администратор может исправлять данные в исключительных ситуациях.

Требование к контролю мест реализуется на уровне сохранения выбора. Перед вставкой записи система проверяет текущую занятость варианта дисциплины и сравнивает ее с лимитом мест. Если свободных мест нет, операция завершается ошибкой. Это правило защищает систему от переполнения дисциплины.

Требование к экспорту данных связано с тем, что результаты распределения должны быть доступны вне приложения. CSV подходит для открытия в табличных редакторах и передачи другим сотрудникам, JSON — для технической проверки и возможных интеграций. В текущей реализации Excel не формируется как отдельный формат, поэтому он не указывается как реализованная функция.

Требования к надежности в рамках прототипа реализованы через транзакции PostgreSQL, уникальные ограничения и контейнерный запуск. Наиболее важные операции, такие как регистрация студента и изменение выбора, выполняются в транзакциях. Это снижает риск частичного сохранения данных при ошибке.

Система должна предотвращать выбор дисциплин студентом, если студент не прошел регистрацию и его Telegram ID не связан с записью в базе данных.

Система должна хранить список доступных групп для каждого выбора отдельно от самих дисциплин, чтобы администратор мог задавать один выбор для нескольких групп.

Система должна показывать студенту заполненность вариантов дисциплин, так как это помогает понимать наличие свободных мест.

Система должна сохранять выборы в централизованной базе данных, чтобы Telegram-бот, REST API и экспорт работали с одним источником истины.

Система должна поддерживать повторный импорт справочных данных, обновляя существующие записи по нормализованным ключам.

Система должна различать источник записи студента, чтобы в итоговых выгрузках было видно, создана запись студентом, администратором или автоматическим алгоритмом.

Система должна позволять администратору исправлять выбор студента вручную, поскольку в реальном учебном процессе возможны исключительные ситуации.

Система должна формировать выходные файлы без необходимости ручного доступа к базе данных.

ГЛАВА 3. ПРОЕКТИРОВАНИЕ СИСТЕМЫ

Для реализации системы выбран монолитный клиент-серверный подход. Все основные компоненты приложения находятся в одном Go-проекте и собираются в один исполняемый файл. При этом внутри кода выделены логические уровни: представление, прикладная логика, доступ к данным, импорт/экспорт и генерация документов. Такой подход соответствует масштабу выпускной квалификационной работы и позволяет снизить сложность разработки и развертывания.

Уровень представления состоит из Telegram-бота и REST API. Telegram-бот является основным интерфейсом для студента: он предоставляет команды регистрации, просмотра доступных выборов, подачи выбора и получения заявления. Для администратора бот предоставляет команды импорта, экспорта, автоназначения и управления администраторами. REST API используется как технический интерфейс для проверки и интеграций.

Уровень прикладной логики реализован в пакете internal/service. Сервис принимает команды от Telegram-бота и REST API, вызывает импорт, обращается к хранилищу, формирует CSV/JSON-выгрузки и инициирует генерацию DOCX. Важной особенностью является то, что сервис зависит от интерфейса Store. Это снижает связность между прикладной логикой и конкретной реализацией PostgreSQL.

Уровень доступа к данным реализован в пакете internal/repository. Он содержит подключение к PostgreSQL через pgxpool, механизм применения миграций, операции импорта, выборки, сохранение записей студентов, проверку ограничений и алгоритм автоматического распределения. Работа с данными выполняется транзакционно в тех местах, где требуется согласованность: регистрация студента, изменение выбора и автоматическое распределение.

[Вставить рисунок 3.1 — Общая архитектура системы]

База данных проектируется вокруг сущностей Student, Choice, ChoiceOption и Enrollment. Студент содержит ФИО, группу и Telegram ID. Choice описывает сущность выбора: обязательную дисциплину по выбору, факультатив или модуль мобильности. ChoiceOption хранит конкретную дисциплину внутри выбора. Enrollment связывает студента с выбранным вариантом дисциплины.

Алгоритм автоматического распределения предназначен для студентов, которые не сделали выбор по required_choice. На вход алгоритм получает код выбора. Система проверяет, что выбор относится к типу required_choice, выбирает студентов доступных групп без записей по этому выбору, получает варианты дисциплин и их текущую заполненность, затем назначает каждого студента на вариант с минимальной заполненностью, если в нем есть свободные места. Алгоритм не хранит и не обрабатывает ранжированные приоритеты студента, поэтому его нельзя называть оптимальным. Корректная формулировка: предложенный алгоритм обеспечивает автоматизированное распределение студентов с учетом доступных мест и текущей заполненности вариантов обязательного выбора.

Вывод по главе: архитектура системы разделяет пользовательские интерфейсы, прикладную логику и работу с данными, но сохраняет простоту монолитного развертывания. Такой подход достаточен для прототипа ВКР и позволяет в дальнейшем развивать отдельные компоненты без полной переработки проекта.

Выбор монолитной архитектуры обусловлен масштабом задачи и характером учебного проекта. Разделение на микросервисы потребовало бы проектирования сетевого взаимодействия между сервисами, отдельного управления несколькими базами данных, мониторинга и более сложного развертывания. Для задачи автоматизации выбора дисциплин такие затраты были бы избыточными на текущем этапе.

При этом монолит не означает отсутствие внутренней структуры. В проекте выделены пакеты, соответствующие разным зонам ответственности. Пакет bot отвечает только за взаимодействие с Telegram Bot API, пакет httpapi — за HTTP-обработчики, пакет service — за прикладные операции, пакет repository — за PostgreSQL, пакет importer — за разбор входных файлов, пакет docx — за генерацию документов.

Сценарий работы студента начинается с команды регистрации. После ввода ФИО и группы система пытается найти студента в базе. Если запись найдена, к ней привязывается Telegram ID. После этого студент может получить список выборов, доступных его группе. Выбор дисциплин выполняется через inline-кнопки, что снижает вероятность ошибки при вводе идентификаторов вручную.

Сценарий работы администратора начинается с подготовки исходных файлов. После загрузки студентов и выборов администратор может контролировать процесс через команды бота или REST API. Команда автоназначения применяется после того, как часть студентов могла не сделать обязательный выбор самостоятельно. В этом случае система распределяет только тех студентов, у которых отсутствует запись по указанному выбору.

Проектирование базы данных учитывает необходимость обновления импортированных данных. Для студентов используются нормализованные ключи ФИО и группы. Для вариантов дисциплин используется нормализованный ключ названия внутри конкретного выбора. Это позволяет повторно импортировать файлы и обновлять существующие записи, а не создавать дубликаты при незначительных различиях форматирования.

Таблица enrollments хранит не только связь студента и дисциплины, но и источник записи. Источник может быть student, manual или auto. Это важно для анализа результата: можно отличить самостоятельный выбор студента, ручную корректировку администратора и автоматическое назначение.

Сущность students: таблица хранит основную информацию о студенте: идентификатор, ФИО, код группы, Telegram ID и временные метки создания и обновления. Наличие Telegram ID позволяет связать запись из импортированного контингента с пользователем Telegram-бота.

Сущность choices: таблица описывает сущность выбора. В ней хранится код выбора, название, тип, образовательная программа, руководитель программы, дедлайн и параметры min_selected/max_selected. Тип выбора ограничен фиксированным набором значений.

Сущность choice_groups: таблица связывает выбор с группами студентов. Благодаря отдельной таблице один выбор может быть доступен нескольким группам, а одна группа может иметь несколько доступных выборов.

Сущность choice_options: таблица содержит конкретные дисциплины внутри выбора. Для каждой дисциплины задаются название, лимит мест, количество зачетных единиц, семестр, преподаватель и ссылка на дополнительную информацию.

Сущность enrollments: таблица фиксирует итоговую запись студента на дисциплину. Помимо связей со студентом и вариантом дисциплины, она хранит источник записи: student, manual или auto.

Сущность admins: таблица хранит Telegram ID администраторов. Она используется ботом для проверки прав доступа к административным командам.

Перед запуском автоматического распределения система получает объект выбора по коду и блокирует его в транзакции. Это снижает риск одновременного изменения данных распределения.

Если тип выбора отличается от required_choice, алгоритм возвращает ошибку. Такое ограничение важно, потому что правила факультативов и модулей мобильности отличаются от обязательного выбора.

Список студентов формируется только из групп, для которых доступен выбранный код. Студенты, у которых уже есть запись по этому выбору, исключаются из автоматического назначения.

Варианты дисциплин загружаются с блокировкой, после чего для каждого варианта определяется текущая занятость. Это значение используется для балансировки распределения.

Для каждого студента варианты сортируются по занятости. Система выбирает первый вариант, где количество занятых мест меньше лимита.

Если все варианты заполнены, студент не получает автоматического назначения. В текущей реализации такая ситуация не приводит к созданию отдельной очереди ожидания.

После успешной вставки записи счетчик занятости варианта увеличивается в памяти, что позволяет учитывать уже назначенных студентов в рамках той же транзакции.

ГЛАВА 4. ТЕХНОЛОГИИ И СРЕДСТВА РАЗРАБОТКИ

Язык Go выбран в качестве основного языка разработки серверной части. Он подходит для создания сетевых приложений, обладает статической типизацией, встроенной поддержкой конкурентного выполнения и простой моделью сборки. Для данного проекта важны предсказуемость компиляции, удобство создания HTTP-сервера и возможность получить один исполняемый файл для контейнеризации.

Для маршрутизации REST API используется библиотека chi. Она предоставляет компактный роутер, поддержку middleware и удобную организацию HTTP-обработчиков. В проекте REST API выполняет вспомогательную роль: через него можно импортировать данные, регистрировать студента, получать выборы и варианты, отправлять выбор, запускать автоназначение и выгружать результаты.

PostgreSQL используется как основная реляционная база данных. Выбор реляционной СУБД обоснован структурой предметной области: студенты, выборы, варианты дисциплин и записи имеют устойчивые связи и ограничения целостности. PostgreSQL позволяет задавать внешние ключи, уникальные индексы, ограничения CHECK и выполнять транзакционные операции.

Для взаимодействия с PostgreSQL используется pgx. В проекте применяется пул соединений pgxpool, что позволяет переиспользовать подключения и выполнять SQL-запросы без дополнительного ORM-слоя. Такой подход сохраняет прозрачность SQL-схемы и хорошо подходит для учебного проекта, где важно явно показать структуру данных и правила работы с ними.

Telegram Bot API выбран как основной пользовательский интерфейс. Этот выбор снижает необходимость разработки отдельного веб-клиента и позволяет студентам использовать уже знакомую среду взаимодействия. Бот поддерживает команды, обычные сообщения, reply-клавиатуру и inline-кнопки для выбора дисциплин.

Docker и Docker Compose используются для воспроизводимого запуска. Dockerfile собирает Go-приложение в отдельном build-слое и переносит бинарный файл в легковесный Alpine-образ. Docker Compose запускает приложение и PostgreSQL, задает переменные окружения и создает volume для хранения данных PostgreSQL.

Форматы CSV и JSON используются для импорта и экспорта данных. CSV удобен для работы с табличными данными и может открываться в офисных приложениях. JSON удобен для интеграций и проверки через REST API. DOCX используется для генерации заявления студента, поскольку этот формат совместим с офисными редакторами и подходит для подготовки документов учебного процесса.

В текущей версии не используются Excel, PDF, Redis, RabbitMQ, Kafka, gRPC, Swagger, Keycloak и отдельные средства мониторинга. Эти технологии встречаются в референсных работах, однако в данном проекте они не реализованы, поэтому не включаются в описание фактической реализации.

Вывод по главе: выбранный стек соответствует задаче разработки компактного клиент-серверного приложения. Go обеспечивает реализацию серверной части, PostgreSQL отвечает за надежное хранение данных, Telegram Bot API предоставляет интерфейс взаимодействия, а Docker Compose обеспечивает воспроизводимый запуск.

Одним из преимуществ Go для данной работы является простая организация зависимостей через go.mod. В проекте явно указаны используемые библиотеки: chi для HTTP-роутинга, Telegram Bot API клиент и pgx для PostgreSQL. Небольшой набор зависимостей упрощает сборку, анализ проекта и объяснение архитектуры в рамках ВКР.

Использование SQL напрямую через pgx позволяет описывать работу с базой данных прозрачно. В дипломной работе это удобно, поскольку можно показать реальные таблицы, миграции, ограничения и запросы. ORM-слой в таком проекте не является обязательным, так как предметная модель компактна и все ключевые операции хорошо выражаются SQL.

Telegram Bot API выбран не как замена серверной архитектуры, а как клиентский канал взаимодействия. Бот не хранит данные самостоятельно, а передает действия пользователя в прикладной слой. Поэтому основная логика остается на сервере и может быть использована как Telegram-интерфейсом, так и REST API.

Docker Compose особенно полезен для демонстрации проекта. Проверяющему или разработчику не нужно вручную устанавливать PostgreSQL, создавать базу и запускать миграции. Достаточно задать переменные окружения и выполнить docker compose up --build. Это повышает воспроизводимость результата и упрощает проверку ВКР.

Формат DOCX выбран для заявления, потому что он является распространенным форматом офисных документов. В проекте документ собирается как OpenXML-пакет. Такой подход демонстрирует понимание структуры формата, хотя в дальнейшем генерацию документов можно вынести в отдельный шаблонизатор или использовать специализированную библиотеку.

ГЛАВА 5. РЕАЛИЗАЦИЯ ПРОГРАММНОЙ СИСТЕМЫ

Реализация проекта организована в соответствии с принятым для Go подходом: точка входа находится в каталоге cmd, а внутренние пакеты размещены в internal. Такая структура позволяет отделить исполняемое приложение от внутренних модулей и сделать границы пакетов более явными.

Точка входа расположена в cmd/app/main.go. На старте приложение загружает конфигурацию, создает пул соединений PostgreSQL, проверяет соединение с базой, применяет миграции, создает слой репозитория и сервисный слой, добавляет администраторов из переменных окружения и выполняет стартовый импорт, если он включен. После этого запускается HTTP-сервер и, при наличии токена, Telegram-бот.

Конфигурация реализована в internal/config/config.go. Поддерживаются переменные HTTP_ADDR, DATABASE_DSN, TELEGRAM_BOT_TOKEN, ADMIN_TELEGRAM_IDS, APP_TIMEZONE, SEED_ON_START, SEED_STUDENTS_FILE и SEED_CHOICES_FILE. Такой подход позволяет запускать одно и то же приложение в локальной среде и в Docker Compose без изменения исходного кода.

Доменные модели находятся в internal/domain/models.go. Основными структурами являются Student, Choice, ChoiceOption и Enrollment. ChoiceType задает три типа выбора: elective, required_choice и mobility. Эти типы используются как в бизнес-логике, так и при проверке ограничений.

Слой прикладной логики находится в internal/service/service.go. Он предоставляет методы импорта файлов, регистрации студента, получения доступных выборов, отправки выбора, административного изменения выбора, экспорта CSV/JSON, генерации DOCX и запуска автоматического распределения. Сервис зависит от интерфейса Store, что упрощает понимание границы между бизнес-логикой и хранилищем.

Работа с PostgreSQL реализована в internal/repository/postgres.go. Функция RunMigrations читает SQL-файлы из каталога migrations, сортирует их по номеру и применяет только те миграции, которые еще не записаны в schema_migrations. Это позволяет развивать схему БД без ручного выполнения SQL-команд.

Импорт студентов выполняется через парсер CSV/JSON. Для студентов используются ФИО, направление и группа. Значение группы нормализуется к формату направление/группа, например 5130904/20102. При некорректном формате возвращается ошибка. Импорт дисциплин группирует строки по коду выбора и создает или обновляет выбор, группы доступности и варианты дисциплин.

Telegram-бот реализует два набора сценариев. Студент может вызвать /register, пройти идентификацию, открыть /choices, выбрать дисциплины через inline-кнопки, посмотреть /my и получить /statement. Для факультативов и модулей мобильности используется множественный выбор с подтверждением в конце. Для обязательной дисциплины по выбору студентский повторный выбор блокируется после первого сохранения.

Администратор может загружать документы с подписями /import_students и /import_choices, просматривать студентов, запускать /auto CODE, выгружать результаты и менять выбор студента командой /set_choice STUDENT_ID CHOICE_CODE OPTION_ID. Также реализованы команды /admins, /add_admin и /remove_admin.

REST API реализован в internal/httpapi/server.go. Он содержит endpoint /health и группу /api. Через REST API можно импортировать студентов и дисциплины, получить список студентов, зарегистрировать студента, получить доступные выборы и варианты, отправить выбор, выполнить административное изменение выбора, запустить автоназначение и выгрузить результаты.

Экспорт CSV/JSON реализован в сервисном слое. Для результатов распределения CSV содержит код выбора, название выбора, название дисциплины, ФИО студента, группу, кредиты, источник записи и дату создания. Отдельно доступны выгрузки всех студентов и зарегистрированных студентов с их выборами.

Генерация DOCX реализована без внешней библиотеки: пакет internal/docx формирует структуру документа OpenXML и упаковывает ее в zip-архив. Для разных типов выбора формируются разные страницы заявления: обязательный выбор, факультативы и модули мобильности. При отсутствии выбранных дисциплин формируется документ с сообщением, что выбранных дисциплин пока нет.

Контейнеризация реализована через Dockerfile и docker-compose.yml. Dockerfile сначала собирает бинарный файл в образе golang:1.26-alpine, затем копирует его в alpine:3.21. В итоговый образ также копируются каталоги migrations и input. Docker Compose запускает PostgreSQL 16 Alpine и приложение, задает healthcheck для базы и публикует порт 8080.

Вывод по главе: программная система реализована как цельное Go-приложение с четким разделением внутренних модулей. Реализованы основные функции, необходимые для автоматизации выбора дисциплин: импорт, регистрация, выбор, распределение, экспорт, генерация документа и контейнерный запуск.

Важной частью реализации является обработка файлов импорта. Парсер студентов поддерживает CSV с разделителем точка с запятой и JSON. Для JSON дополнительно предусмотрена нормализация JSON-подобного содержимого: заменяются значения NaN и лишние завершающие запятые. Это полезно при работе с файлами, полученными из внешних источников, где формат может быть не строго валидным.

Парсер выборов извлекает набор полей, необходимых для создания выбора и вариантов дисциплин. Поддерживаются русскоязычные и англоязычные названия колонок, например deadline и дедлайн, group_codes и группы, option_title и дисциплина. Это делает импорт более устойчивым к различиям в заголовках входных файлов.

При сохранении выбора студента используется функция ReplaceStudentChoiceEnrollments. Она блокирует студента и выбор в транзакции, проверяет дедлайн, доступность группы, принадлежность вариантов к выбору, правила валидации и свободные места. После этого старые записи по данному выбору удаляются, а новые добавляются. Для обязательного выбора повторная студенческая отправка блокируется до удаления старой записи.

Для факультативов после последних изменений реализовано правило отсутствия лимита по количеству выбранных дисциплин. В интерфейсе бота факультативы работают как множественный выбор: студент отмечает несколько вариантов, затем нажимает кнопку подтверждения. На уровне валидации значения min_selected и max_selected для elective не ограничивают количество выбранных вариантов.

Для модулей мобильности используется проверка суммы зачетных единиц. Система суммирует credits выбранных вариантов и сравнивает результат с min_selected и max_selected. В текущей модели эти поля для mobility трактуются как границы суммы кредитов, а не как количество дисциплин.

Автоматическое назначение required_choice использует простую стратегию балансировки по заполненности. Перед назначением варианты сортируются по количеству уже занятых мест. Если у нескольких вариантов одинаковая заполненность, используется порядок по идентификатору. После успешной записи счетчик занятости варианта увеличивается в памяти, чтобы следующий студент учитывал уже сделанное назначение.

В REST API ошибки доменного уровня возвращаются как HTTP 400, а отсутствие сущности — как HTTP 404. Такой подход достаточен для прототипа и позволяет клиенту отличить неверный запрос от ситуации, когда студент или выбор не найден. Для промышленной версии можно расширить структуру ошибок кодами и машинно-читаемыми причинами.

При экспорте CSV используется разделитель точка с запятой, что удобно для русской локали и открытия файла в табличных редакторах. JSON-экспорт формируется с отступами, чтобы результат можно было читать вручную при проверке. Оба формата создаются в памяти и отдаются через REST API или отправляются администратору как файл в Telegram.

Модуль cmd/app содержит точку входа приложения. Этот модуль связывает конфигурацию, базу данных, миграции, сервисный слой, REST API и Telegram-бота в единый процесс.

Модуль internal/config отвечает за чтение переменных окружения и преобразование строковых значений в типы Go, например список Telegram ID администраторов или булев флаг автоимпорта.

Модуль internal/domain содержит структуры предметной области. Эти структуры используются одновременно в репозитории, сервисном слое, REST API и экспорте JSON.

Модуль internal/service реализует прикладные операции и служит центральной точкой бизнес-логики. Он не знает деталей HTTP или Telegram, поэтому может использоваться разными интерфейсами.

Модуль internal/repository содержит SQL-запросы, миграции и правила сохранения данных. Здесь реализованы проверки дедлайна, группы, мест, типа выбора и автоматическое распределение.

Модуль internal/bot реализует пользовательский и административный сценарии Telegram-бота. Пакет обрабатывает команды, сообщения, документы и inline-кнопки.

Модуль internal/httpapi реализует REST API. Этот интерфейс полезен для проверки, интеграции и получения файлов без использования Telegram.

Модуль internal/importer разбирает входные CSV/JSON-файлы и приводит их к структурам импорта. Здесь реализована валидация формата группы и дедлайна.

Модуль internal/docx формирует заявление студента в формате DOCX. Документ создается как OpenXML-пакет и содержит страницы по выбранным дисциплинам.

Endpoint GET /health: используется для проверки доступности HTTP-сервера. В успешном случае возвращает JSON со статусом ok.

Endpoint POST /api/import/students: принимает файл студентов и запускает импорт через сервисный слой. Поддерживается multipart-загрузка и raw upload с именем файла.

Endpoint POST /api/import/choices: принимает файл выборов и дисциплин. После разбора данные сохраняются в таблицы choices, choice_groups и choice_options.

Endpoint POST /api/students/register: регистрирует студента по Telegram ID, ФИО и группе. Метод используется как REST-аналог регистрации через Telegram-бота.

Endpoint GET /api/students/{id}/choices: возвращает выборы, доступные группе конкретного студента.

Endpoint GET /api/choices/{code}/options: возвращает варианты дисциплин выбранного кода, включая текущую занятость.

Endpoint POST /api/students/{id}/choices/{code}/submit: сохраняет выбор студента и применяет студенческие ограничения, включая дедлайн.

Endpoint POST /api/admin/students/{id}/choices/{code}/submit: позволяет администратору вручную изменить выбор студента без проверки дедлайна.

Endpoint POST /api/choices/{code}/auto-assign: запускает автоматическое распределение студентов без выбора для required_choice.

Endpoint GET /api/export/results: формирует CSV или JSON с результатами распределения.

Endpoint GET /api/students/{id}/application.docx: формирует DOCX-заявление выбранного студента.

Переменная HTTP_ADDR: адрес, на котором запускается HTTP-сервер. В Docker Compose используется значение :8080.

Переменная DATABASE_DSN: строка подключения к PostgreSQL. В контейнерной среде указывает на сервис postgres внутри Docker-сети.

Переменная TELEGRAM_BOT_TOKEN: токен Telegram-бота. Если значение пустое, приложение запускает только REST API.

Переменная ADMIN_TELEGRAM_IDS: начальный список Telegram ID администраторов, разделенный запятыми.

Переменная APP_TIMEZONE: часовой пояс приложения. По умолчанию используется Europe/Moscow.

Переменная SEED_ON_START: флаг автоматического импорта стартовых данных после миграций.

Переменная SEED_STUDENTS_FILE: путь к файлу студентов для стартового импорта.

Переменная SEED_CHOICES_FILE: путь к файлу дисциплин для стартового импорта.

ГЛАВА 6. ТЕСТИРОВАНИЕ СИСТЕМЫ

Цель тестирования состоит в проверке работоспособности основных сценариев и корректности отдельных правил бизнес-логики. В проекте присутствуют автоматические модульные тесты, а также была выполнена ручная проверка запуска приложения через Docker Compose.

Модульные тесты импортера проверяют разбор JSON-подобного файла студентов, разбор примера дисциплин и обработку некорректного формата группы. Это важно, поскольку импорт является начальным этапом работы системы: без корректной загрузки студентов и дисциплин дальнейшие сценарии невозможны.

Тесты правил в репозитории проверяют нормализацию группы, валидацию обязательного выбора, отсутствие ограничений по количеству факультативов и проверку суммы зачетных единиц для модулей мобильности. Эти проверки отражают ключевые предметные правила системы.

Команда запуска тестов в проекте: go test ./... . В текущей среде использовалась команда GOCACHE=/tmp/gocache GOTOOLCHAIN=auto go test ./..., так как локальный кэш Go находился в read-only области, а версия Go подтягивалась через toolchain.

ГЛАВА 7. РЕЗУЛЬТАТЫ И ПЕРСПЕКТИВЫ РАЗВИТИЯ

В результате выполнения работы разработан программный прототип клиент-серверного приложения для автоматизации выбора и распределения студентов по дисциплинам по выбору. Реализованы серверная часть на Go, база данных PostgreSQL, Telegram-бот, REST API, импорт и экспорт данных, генерация DOCX-заявлений и запуск через Docker Compose.

Разработанная система соответствует основным требованиям, сформулированным для прототипа. Студент может зарегистрироваться и выбрать дисциплины, администратор может загрузить исходные данные, управлять выбором, запускать автоматическое распределение и выгружать результаты. Данные сохраняются в PostgreSQL, а миграции выполняются автоматически при старте.

Текущая версия имеет ограничения. В ней отсутствует отдельный веб-интерфейс, не реализованы Excel-импорт и Excel-экспорт, не выполняется генерация PDF, не хранится ранжированный список приоритетов студента, а автоматическое распределение применяется только для обязательного выбора. Эти ограничения не мешают использовать систему как прототип, но важны для честной оценки результата.

Возможные направления развития включают создание веб-интерфейса администратора, расширение REST API и документирование его через OpenAPI, добавление Excel-экспорта, генерацию PDF-документов, хранение приоритетов студентов, расширение алгоритма распределения и добавление интеграционных тестов с PostgreSQL и Telegram Bot API.

Практическая значимость работы состоит в том, что разработанная система демонстрирует применимость Go, PostgreSQL и Telegram Bot API для автоматизации реального учебного процесса. Проект может быть использован как основа для дальнейшей разработки и адаптации под регламенты конкретной образовательной организации.

С точки зрения учебного результата проект демонстрирует полный цикл разработки: анализ задачи, проектирование данных, реализацию серверной логики, создание пользовательского интерфейса, контейнеризацию, тестирование и подготовку материалов для эксплуатации. Это позволяет рассматривать систему не как набор отдельных функций, а как законченный программный прототип.

Перспектива добавления веб-интерфейса особенно актуальна для администратора. Через веб-панель можно было бы просматривать заполненность дисциплин, фильтровать студентов, запускать распределение и скачивать файлы без использования команд Telegram. При этом существующий сервисный слой и REST API могут стать основой для такого интерфейса.

Расширение алгоритма распределения может включать учет приоритетов студента. В этом случае модель данных должна хранить не только факт выбора, но и порядок предпочтений. Алгоритм должен будет учитывать лимиты мест и стремиться удовлетворить более высокие приоритеты. Такая задача сложнее текущей и требует отдельного проектирования.

Еще одним направлением является развитие отчетности. В текущей версии доступны CSV и JSON, а DOCX используется для заявления. Для практического применения может быть полезен Excel-экспорт с отдельными листами по группам и дисциплинам, а также генерация PDF-версий заявлений для архивного хранения.

Для повышения надежности можно добавить журналирование административных действий, более подробные коды ошибок и резервное копирование базы данных. Эти функции не являются обязательными для прототипа, но важны при переходе к эксплуатации в реальном учебном процессе.

ЗАКЛЮЧЕНИЕ

В ходе выполнения выпускной квалификационной работы была достигнута поставленная цель: разработан программный прототип клиент-серверного приложения для автоматизации распределения студентов по дисциплинам по выбору с использованием языка Go.

Были выполнены основные задачи работы: проведен анализ предметной области, определены требования к системе, спроектирована архитектура приложения и структура базы данных, реализованы серверная часть, Telegram-бот, REST API, импорт и экспорт данных, генерация DOCX-документов, механизм автоматического распределения и контейнерный запуск через Docker Compose.

Разработанная система позволяет сократить ручную обработку заявок, централизовать хранение данных, контролировать ограничения и формировать итоговые выгрузки. При этом текущая версия честно ограничена рамками прототипа: часть функций, таких как веб-интерфейс, Excel-экспорт, PDF-генерация и расширенное интеграционное тестирование, может быть реализована в дальнейшем.

Таким образом, результат работы подтверждает возможность применения выбранного стека технологий для решения задачи автоматизации выбора дисциплин в образовательной организации.
